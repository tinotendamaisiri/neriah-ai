# Data Models

See `backend/shared/models.py` for the canonical Pydantic definitions.
This document explains partition key choices and query patterns.

## Cosmos DB Containers

### `teachers` — partition key: `/phone`
Phone number is the primary identity for WhatsApp teachers.
Single document per teacher. Looked up on every inbound WhatsApp message.

**Common queries:**
- `SELECT * FROM c WHERE c.phone = @phone` (point read, PK = phone)

---

### `classes` — partition key: `/teacher_id`
All classes for a teacher are in the same partition.
Supports fast "list all classes for teacher" queries without cross-partition scan.

**Common queries:**
- `SELECT * FROM c WHERE c.teacher_id = @teacher_id` (single partition)

---

### `students` — partition key: `/class_id`
All students in a class are co-located.
Supports fast "list all students in class" queries.

**Common queries:**
- `SELECT * FROM c WHERE c.class_id = @class_id` (single partition)

---

### `answer_keys` — partition key: `/class_id`
Answer keys are keyed by class. A class may have multiple answer keys (one per subject/assessment).

**Common queries:**
- `SELECT * FROM c WHERE c.class_id = @class_id` (single partition)
- Point read by `(id, class_id)` when loading a specific key for grading

---

### `marks` — partition key: `/student_id`
All marks for a student are co-located.
Supports per-student history queries efficiently.

**Common queries:**
- `SELECT * FROM c WHERE c.student_id = @student_id ORDER BY c.timestamp DESC`
- `SELECT * FROM c WHERE c.student_id = @student_id AND c.answer_key_id = @answer_key_id`

**Note:** Querying all marks for a class (for analytics) requires a cross-partition query
across all student IDs in that class. For MVP this is acceptable; at scale, consider a
materialized view updated by a Cosmos change feed trigger.

---

### `sessions` — partition key: `/phone` | TTL: 86400s
WhatsApp conversation state. One document per active teacher.
Auto-deleted by Cosmos TTL after 24 hours of inactivity.
Session is recreated fresh when teacher sends next message.

**Common queries:**
- Point read by `(phone, phone)` — O(1) on every inbound message

---

## Document ID Strategy

- All IDs are UUID4 strings generated by `uuid.uuid4()` at create time.
- Exception: `sessions` — `id` = `phone` (no UUID, one doc per phone, upserted in-place).

## Schema Evolution

Pydantic models use `Optional` with defaults for all non-required fields.
Adding new optional fields to a model is backwards-compatible with existing Cosmos documents
(Pydantic will use the default value if the field is missing from the stored document).

Removing fields or changing field types requires a migration script.
